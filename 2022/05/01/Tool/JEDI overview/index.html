JEDI-overview

Posted on 2022-05-01 Edited on 2025-10-05 In Tool Views:

来源：https://jointcenterforsatellitedataassimilation-jedi-docs.readthedocs-hosted.com/en/latest/overview/index.html

What is JEDI

准确的天气预报离不开两大“原料”：一是对当前和过去大气、海洋状态的全面观测；二是能把这些观测“吃进”去、并算出未来几分钟到几天变化的可靠计算机模式。

JEDI（Joint Effort for Data assimilation Integration）正是把这两样原料放到“数据同化”这口大锅里翻炒的系统。它的长远目标，是给研究和业务提供“一套通用、不挑模式、不挑分量”的数据同化框架，让社区少做重复活儿，把“研究→业务”的跑道缩到最短。

JEDI 是社区共建的开源项目，代码托管公开，欢迎全球科研力量“拎包入住”。
项目组织里，开发者、评审员、治理机构各负其责，互动流程有章可循，保证大家并行不悖、持续高效地往前滚代码。

Who Uses JEDI?

一个可靠高效的数据同化系统，是任何“地球系统预测”框架的标配；因此，JEDI 的潜在用户远不止业务中心——

  • 政府机构
  • 大学
  • 私营公司

凡是做天气预报的，都是它的“客户群”，其中就包括 JCSDA 的“老搭档”：美国国家海洋和大气管理局（NOAA）、美国国家航空航天局（NASA）、美国海军和美国空军。

不过，JEDI 可不只是预报员的“专利”。现代科学的一条铁律是：“观测说了算”。任何理论、任何新模型，都得靠“预测 vs 观测”的 PK 来验真。于是，数据同化也成了科研“试金石”——

  • 新的物理、化学、生物模型
  • 新的同化算法
  • 新的观测算子

都要在这里过一遍秤。JEDI 因此成了全球科研人员的大试验场，不管你是高校、政府还是企业，都能进来“搭台唱戏”。

更妙的是，JEDI 天然自带“双向八车道”：

  • 研究→业务（R2O）：学者在同一份代码里验证新模块，业务中心直接拿来用，无需“二次搬家”
  • 业务→研究（O2R）：业务里跑出更准的算法，学术界马上拿回去拆解成功秘诀

硬件方面，JEDI 同样“上不封顶、下不挑嘴”：

  • 世界顶尖超算 + 最复杂的大气-海洋模式：能跑
  • 笔记本：也能跑（自带理想化模型、低分辨率配置，教学/玩票都够用）

人才培养方面：
天气与海洋研究的未来，握在今天研究生手里；而数据同化这门“微妙手艺”，必须边玩边学才能精通。JEDI 提供了交互式环境，让学生和老师在动手实践中：

  • 拆解同化原理
  • 体验地球系统预测的完整流程

把课堂直接搬进“预报一线”。

Why Should I Use JEDI?

现代软件开发是“团战”：

  • 分散在不同城市、时区的开发者先各自写功能，再合流集成
  • 成员可能素未谋面，却必须把“撞车”和“返工”降到最低

过去二十年，软件行业靠两大法宝搞定这一难题：

  1. 敏捷方法（如 Scrum）

    • 早交付、常交付
    • 团队自组织
    • 随时响应用户需求

  2. 云平台（如 GitHub）

    • 让代码跨洲托管、秒级同步
    • 合并、回滚、评论一条龙

JEDI 直接把这套“工业级标配”搬进天气预报圈：

  • 开发环境 = GitHub 全家桶

    • git：版本控制
    • GitHub：在线协作与分发
    • ZenHub：敏捷看板 + 需求/缺陷跟踪
    • 三者连成闭环，新人五分钟即可上手，权限依旧可层层把关

  • 代码设计 = 面向对象 + 泛型编程

    • 用抽象类、模板搭骨架
    • 新模式、新算法、新观测往里插，无需大动干戈

结果：

  • 一套框架同时服务研究侧与业务侧
  • 全球开发者并行开干，却共用同一条“主线”
  • 系统永不过时，持续为大气与海洋科学界输出统一、创新、可扩展的预报能力

JEDI High Level Requirements

地球观测系统一直在升级，新卫星、新仪器接连上天，要想“吃干榨尽”这些数据，就得让软件也跟着“常换常新”。数据量年年翻倍，同化算法必须越跑越快、越省越巧；而新一套观测系统动辄几亿甚至几十亿美元，谁都不想等五年十年才用上。JEDI 的其中一个目标，就是给研究人员搭一条“快车道”，新点子一出，就能迅速从实验室滑进业务值班室。

天气预报和环境预报也不再只盯着大气，还要把海洋、陆面、海冰、大气化学、生物圈统统拉进来，做成真正的“地球系统模式”。JEDI 必须能跟这些“新成员”打交道，让模式从“单打独斗”变成“全队配合”。

观测和模式在变，同化算法自己也在“卷”。JEDI 会把当今业务里最好用的算法先搬进来，再留好“插座”，谁有新想法——机器学习、多尺度混合、非高斯处理——都能随时插上去试。注意，“统一框架”可不是“一刀切”；各家机构照样可以在里面做自己的“私房菜”，互不耽误。

后台的超算越来越“豪华”，CPU、GPU、众核、异构一起上，可要是跑不满，电费照样烧。社区里大家都在琢磨怎么让模式和同化系统“能伸能缩”，把几十万核用到极致。JEDI 从第一天就把“可扩展性”写进蓝图，换了新机器也能快速“拎包入住”。

研究要“灵活”，业务要“皮实”，JEDI 得两头顾：一边让学生、老师能在笔记本上玩得起、学得会；一边让值班员能在巨型机上 7×24 小时稳、准、快地跑，还要符合编码规范、好维护、好交接。

JEDI General Methodology

JEDI 要满足的需求五花八门，而且很多本身就够复杂，合在一起更是“难上加难”。现代软件工程对付这种庞然大物的核心办法是“分而治之”——把大问题拆成小块，让不同团队各管各的，互不干扰、互不踩脚。科学家也省心，只盯自己那一亩三分地，不用把整个系统都啃下来。面向对象等现代技术不仅把“分”做成自动强制执行，还保证全球分布的团队能在同一份代码上并行开工，就像在一个办公室一样。

为了把这场“跨国接力”跑顺，JEDI 把现代工具全套配齐：版本控制、需求/缺陷跟踪、自动化回归测试，一条龙到位；信息怎么提交、怎么同步、怎么回溯，都有现成脚本和模板，跨机构开发、长期维护都能用得上。

工具和人就位后，第一件事就是“画插座”——给各模块定义通用、抽象的接口。这套接口长什么样，会充分借鉴 ECMWF 的 OOPS 等前辈经验。接口一旦定版，就能在上面搭“万能插头”式的高层应用，先不管底层是大气模式还是海洋模式，都能插进来用。

接口有了，老代码也不用“一刀切”重写。哪部分ready，就把哪部分先包上新接口；老程序改几行调用语句，就能继续跑，系统始终“不断电”。当然，会先挑一批“最急缺”的接口优先落地，让部分应用提前尝鲜，不用等全部完工。

最后一条底线：对业务中心来说，换上这套新框架不能“卡脖子”，也不能让跑分掉速——该快的还得快，该稳的还得稳。

Governance of a Community System

要不要把一段代码收进“社区大仓库”，首先看它对社区有没有用——“有用”是硬杠杠，科学和技术质量只是门槛。
在 JEDI 这种社区同化-预报系统里，谁拍板、怎么拍板，必须写得明明白白：项目内部负责人、JCSDA 管理层、社区监督委员会，各投什么票、负什么责，都在治理章程里写清楚。

评审环节只管“质量够不够”；万一最后没让进，不代表你的科学不优秀，只说明它暂时只有你自己用。这样的“私家定制”代码可以留在独立仓库里，构建系统会自动按需捞取，绝不拖累别人。要做到这点，全靠现代软件架构——老 Fortran 时代根本玩不转。

当年用 F77 写大系统，想把“私人补丁”摘出去，就得留一堆空壳子哑函数，全局变量还常常“掉线”。代码越攒越多，空壳子四处飞，很快就成“蜘蛛网”。最后大家索性“一把梭”全塞进主仓库，结果系统臃肿、编译巨慢、人人喊打。

如今有了模块化、泛型编程、关注点分离这些“新式武器”，想带就带、想甩就甩，既不会掉链子，也不会把公共仓库撑成“胖子”。私人代码、社区代码各回各家，构建系统一键搞定，干净利索。

Structure of JEDI

在老式代码里，继承树上的一个专用子类被删掉时，就像撕掉一页便利贴——撕完不留痕；而在另一极端，由一堆对象拼成的高层应用，也不会因为别的应用用了同样的对象就“踩到脚”。软件包有的只让底层“打补丁”（像浏览器或 Photoshop 的插件），有的只让高层“搭积木”（如系统库、MPI），也有的两头都开放——JEDI 就属于这种“两头通吃”的类型。

但这一切有个大前提：中间层的接口必须“稳如老狗”。

  • 一旦基类接口改一行，所有子类都得跟着动；
  • 一旦高层应用依赖的对象接口变脸，所有相关应用都要返工。

如果这种“拆东墙”的事年年有、月月有，系统立刻就会被人嫌弃。因此，社区统一同化-预报系统的治理框架第二件事就是：

  1. 让接口变动“少而精”，每次改都必须“师出有名”；
  2. 改之前写清楚理由，改之后广而告之，谁受影响一目了然。

第三件事是“谁有钥匙”。

  • 代码每一层，尤其是发版前的“临门一脚”，必须明确谁能审、谁能拍板。
  • 通常做法是给每个核心模块设一个小型“守门员”池，并规定 PR（Pull Request）至少被几位守门员点赞才能合进去。
  • 人数既要够“挑刺”，又不能多到“拖堂”；守门员可信，就能临时拉人助阵或把接力棒递出去——小修小改不必兴师动众。

JEDI Roles and Responsibilities: Our Governance Framework

Management Oversight Board (MOB)

JCSDA 的管理与监督委员会（MOB）由 JCSDA 各伙伴机构以及更广泛社区的代表组成。
它的“权力清单”包括：

  1. 拍板 JCSDA 的大政方针、目标、优先级，推荐哪些观测系统将来可以上业务；
  2. 确保各赞助机构、高校、公司之间“不拆台、多补台”；
  3. 万一社区正常讨论“翻车”达不成共识，MOB 出面一锤定音，避免项目卡壳。

日常层面，MOB 负责把握项目的“大方向、大愿景”——跟谁战略合作、立什么规矩、钱和人怎么管，都属于他们的望远镜范围。只要社区扯皮太久没结果，MOB 就升级成“最终裁判”，把高阶决策迅速落地。

Executive Team (ET)

JCSDA 执行团队（ET）由几位副主任组成，与 JCSDA 主任一起办公。
他们的核心任务：

  • 让 JEDI 项目“活得久、活得好”，技术上不掉链子，社区里不掉人气。
  • 给主任和 MOB 当“参谋”，一起划定项目的大方向、大合作、大政策。
  • 凭借对软件与服务的“老司机”级理解，给新上路的贡献者当“导航”，少走弯路。

Project Leads

JEDI 项目的“一线掌柜”是各位项目负责人（Project Leads），包括：

  • 总项目负责人（JEDI Project Leader）
  • 下设四条线的小组长：软件基础设施、模式-观测接口、数据同化、云基础设施

他们得按《JCSDA 软件治理与监督文件 Rev 1》的“家规”办事，和 JCSDA 主任、副主任以及“投了真金白银”的伙伴机构保持同频。

具体活儿清单：

  1. 把 JEDI 软件按时按点交到各方“甲方”手里；
  2. 统筹“正规军”（核心团队）和“志愿军”（贡献代码的伙伴）的力量，依据 JCSDA 年度运行计划（AOP）和各自任务书干活；
  3. 写需求、回邮件、拉合作、找重复、减冗余、拉新伙伴、帮旧伙伴——一条龙；
  4. 每季度向 MOB 和 ET 交“作业”，并组织代码冲刺、培训营。

在代码层面，他们同时兼三职：

  • 贡献者——自己写代码；
  • 评审者——给别人挑刺；
  • 管理员——管钥匙、合并 PR。

合并红线：

  • 必须有人背书的科学质量、技术质量；
  • 必须通过所有测试；
  • 必须符合编码规范和文档要求；
  • 遵循“do-no-harm”——不踩坏主干。

一句话：代码没拿到“合格章”，谁也不能合进主分支。

Contributors

  • 贡献者：只要交过代码、文档、设计或其他成果，都算；一旦 PR 被合并，就升级为“开发者”。

    • 包括：JCSDA 核心员工、合作方“抵现”人员、外部个人或承包商。
    • 做事方式：在 GitHub 上提 PR、审 PR、开 issue、公开讨论，全程透明。

  • 开发者义务：

    1. 写清楚文档；
    2. 保证自己的分支跟 develop 分支同步；
    3. 自测通过再提 PR；
    4. 互相 code review，没人能“只写不看”。

  • 用户：人数最多的群体。
    贡献者归根到底是为用户打工，因此我们尽量把“提 PR”的门槛踩平——今天你会用，明天你就能改，后天你就是开发者。
